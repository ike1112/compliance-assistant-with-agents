## Problem Statement

Six phases of production-hardening shipped (Phase 1 build complete; HUMAN-GATE
deploy outstanding). README and ARCHITECTURE.md rewritten the prior turn to
surface the work. But the surfacing was anchored to the WA-Lens + IaC narrative,
which is the right wedge for *platform / cloud engineer* audiences and the
wrong wedge for the actual stated target — **ML / AI Platform / Applied AI
engineer** roles. For that audience, the strongest signal in the repo is buried
as one row in the README's status table.

## What Makes This Work (the actual wedge)

The Phase 3 RAG evaluation harness, specifically the **author-provenance +
frozen-gold-set mechanism**, is genuinely uncommon in ML/AI platform candidate
portfolios. Most candidates show eval *results* (Ragas scores, BLEU/ROUGE,
side-by-side comparisons). This repo shows an eval *system* with a structural
defense against the "your AI graded its own homework" critique:

- Gold set authored by codex, not by the project's own ralph autonomous loop
  (declared in [`tests/evals/gold/PROVENANCE.md`](../../tests/evals/gold/PROVENANCE.md)).
- A frozen-gold test ([`tests/evals/test_gold_frozen.py`](../../tests/evals/test_gold_frozen.py))
  that fails if the judged diff touches the labels.
- LLM-as-judge with a committed rubric (not an opaque scoring call).
- A deterministic offline gate (`pytest tests/evals -m gate -q`) recorded
  against fixtures so the eval bar runs in CI with no Bedrock spend.
- The chunking decision is driven by *data* the harness produces — Phase 3
  scores ≥ 2 configs, writes [`tests/evals/report.md`](../../tests/evals/report.md),
  and a test asserts `infra/cdk.json` context matches the winner. The
  infrastructure reads what the eval picked.

That last clause is the actual signal: data-driven infrastructure decisions,
not vibes. Most candidates can't show that link.

Supporting context (IaC, observability, review gate, WA-Lens audit) proves the
eval harness can run in production. It is not the headline.

## Constraints

- Target role is ML/AI Platform / Applied AI engineer (confirmed Q3).
- No one in the target audience has seen the artifact yet (confirmed Q1).
- The just-written README + ARCHITECTURE.md lead with WA-Lens + IaC, not the
  eval harness.
- Approach selected: **Cathedral** (C) — reframe + external write-up +
  Phase 1 deploy + evidence. Sequenced, not parallel.

## Premises

1. **Target audience** = ML/AI Platform / Applied AI engineer roles, not
   general platform/cloud roles, not staff-level posture roles.
2. **Strongest signal** = Phase 3 eval harness, specifically the
   author-provenance + frozen-gold mechanism. The IaC / observability /
   methodology / meta-quality-gate work is supporting context.
3. **Current surfacing docs are wrong-shaped** for the target audience —
   they lead with WA-Lens phases + IaC; the eval-harness story is one row
   in a status table.
4. **No audience contact has happened.** Zero ML/AI platform engineers,
   hiring managers, or recruiters have seen and reacted to this artifact.
5. **Cathedral approach is sequenced, not parallel.** Audience contact
   happens after reframe but before write-up + deploy, so feedback shapes
   what gets written about and what evidence gets captured.

## Approaches Considered

### Approach A: Minimal viable — reframe + audience contact
Re-point README + ARCHITECTURE.md to lead with the eval harness; send
to 2-3 specific people in target audience; iterate on reaction.
S effort, low risk, cheapest signal-per-hour.

### Approach B: Ideal architecture — reframe + portable external write-up + audience contact
All of A plus a standalone technical write-up (blog / dev.to / long-form
GitHub README in a separate repo) focused on the eval harness mechanism.
Portable signal independent of "will they clone the repo." M effort, low risk.

### Approach C (chosen): Cathedral — reframe + write-up + Phase 1 deploy + evidence
All of B plus the billable Phase 1 deploy: real AWS, CloudWatch screenshots,
KB ingestion proof, real generated report with Sources block, embedded as
evidence. L effort, medium risk (real AWS state + costs + drift), closes
the "synthesized not deployed" credibility gap concretely.

## Recommended Approach (sequenced)

**Phase 1 of execution: Reframe the surfacing docs (2-3 hours CC)**

The eval harness becomes the lede. Concrete changes:

- README opening reframed: lead with "I built a RAG eval harness with
  author-provenance + frozen-gold + LLM-as-judge + offline deterministic
  gate; the harness picks the chunking value the IaC reads." Move WA-Lens
  / IaC narrative to "supporting" framing.
- ARCHITECTURE.md restructured: Phase 3 promoted to its own headline
  section ahead of the phase-by-phase narrative. The phase-by-phase becomes
  "Other phases that support the eval-harness story in production." Phase 3
  section expanded with the structural-defense argument (why the
  provenance + frozen-gold check is the actual signal).
- Add a "How to verify the eval-harness claims" quick-start: 3 commands
  someone can run to see the gold-set provenance check, the frozen-gold
  test, and the offline gate exit 0.

**Phase 2 of execution: Audience contact, round 1 (within ~1 week)**

Identify and contact **2-3 specific people** in the target audience.
Concrete profile: ML/AI platform engineer or applied AI engineer at a
company you'd actually want to work at, not a general "engineer friend."
The ask is specific: *"I'm building a portfolio piece around production
RAG and would value 15 minutes of your reaction — specifically, what's
the first thing you noticed and what would make you want to dig in
further?"*

Record what they react to (notes immediately after). The reaction shapes
Phase 3 and Phase 4.

**Phase 3 of execution: External technical write-up (1-2 days human work
/ ~half day CC for draft)**

Standalone artifact in the format the target audience reads. Either
(a) a long-form GitHub README in a separate `compliance-assistant-eval-harness`
repo with the harness factored out as a reusable pattern, or (b) a blog
post. Content is shaped by what Phase 2 contacts reacted to. Don't write
this before Phase 2 — you'll write the wrong thing.

**Phase 4 of execution: Phase 1 deploy + evidence capture (1-2 days human +
billable AWS, time after Phase 2 + Phase 3 to ensure you capture the right
evidence)**

Run `cdk bootstrap` + `cdk deploy` per the PRD HUMAN-GATE. Capture:

- CloudWatch dashboard screenshot showing the SLO-anchored alarms (the
  ones synthesized from `docs/SLOs.md`)
- KB ingestion job COMPLETE evidence
- A real generated `output/2-report.md` with a non-empty `## Sources`
  block, committed to the repo as `examples/deployed-report.md`
- A short `DEPLOYED_EVIDENCE.md` summarizing what was captured and when

Plan teardown: schedule the deploy close to a 1-2 week window where
you're actively job-applying, so the screenshots are recent. Tear down
afterwards to stop the bill.

## Success Criteria

- **Phase 1 of execution success:** README first 30 seconds (above the
  first table) names the eval harness as the lede; the author-provenance
  + frozen-gold mechanism is findable without scrolling past unrelated
  content.
- **Phase 2 of execution success:** ≥ 2 of 3 contacts react to the
  eval harness specifically (not just "looks nice"). The reaction is
  specific enough to inform Phase 3.
- **Phase 3 of execution success:** External write-up exists at a public
  URL you can paste into recruiter conversations. Length and depth shaped
  by what Phase 2 contacts asked about.
- **Phase 4 of execution success:** Repo has `examples/deployed-report.md`
  with a real Sources block, a CloudWatch dashboard screenshot in
  `docs/analysis/_evidence/`, and a `DEPLOYED_EVIDENCE.md` summary.
- **Overall success:** ≥ 1 inbound or expansion-of-conversation that
  cites this artifact concretely within 4 weeks of completion. That's
  the actual signal — anything less is interest, not demand.

## Distribution Plan

- GitHub repo with the reframed README as the entry point
- External write-up at a public URL (blog / GitHub / dev.to)
- Both linked from your resume, LinkedIn, and pasted into recruiter
  chats and cover letters when applying for the target roles

No package / binary / container distribution needed — the artifact is
the writeup + the repo, not a deployable software product.

## What I noticed about how you think

- You picked **"both"** when offered portfolio-validation vs
  product-validation, and **"C) Cathedral"** when offered minimal /
  ideal / cathedral. There's a consistent pattern: when offered a
  recommended-minimal option, you go bigger. That's not wrong, but it's
  worth naming — the lake principle is "boil it when AI makes it
  cheap," not "boil it because more is better." Phase 1 deploy in
  particular is not AI-cheap; it's real-money expensive.
- You delegated the premise check ("you reason and decide") rather
  than engaging each premise. That's trust, which is fine, but the
  premise check exists specifically to catch a hidden disagreement
  before it becomes a built-the-wrong-thing problem. Worth flagging
  in case you do disagree with one of the five.
- You said "are you done with all the phases" earlier — that question
  is the most important signal in this session. It's the moment a
  builder pauses and asks "now what?" without yet knowing the
  reframe is the answer. The instinct to check before continuing is
  the right instinct.
- You took **C (the most expensive Cathedral path)** even after I
  recommended A. Honor it, but watch for the failure mode: spending
  3-5 days on B + C before getting any audience reaction means
  shaping the wrong write-up and capturing the wrong evidence. The
  sequencing in the recommended approach is the guard against that.
