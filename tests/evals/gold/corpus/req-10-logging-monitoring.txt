PCI DSS Requirement 10: Log and Monitor All Access to System Components and Cardholder Data

Scope and Intent
Requirement 10 ensures that activity in the cardholder data environment can be reconstructed and monitored. Logs support incident detection, investigation, and accountability. Logging must cover users, administrators, system events, security events, and access to cardholder data.

Audit Log Coverage
Audit logs should capture enough detail to show who did what, when it happened, whether it succeeded, and where it occurred. The requirement is not satisfied by collecting logs without meaningful fields or by logging only a narrow subset of in-scope systems.

PCI DSS v4.0 Requirement 10.2.1 requires audit logs to be enabled and active for all system components and cardholder data. The logs must capture events needed to support anomaly detection and forensic analysis.

PCI DSS v4.0 Requirement 10.2.1.1 requires audit logs to capture all individual user access to cardholder data. This supports accountability when stored or processed cardholder data is viewed or used.

PCI DSS v4.0 Requirement 10.2.1.2 requires audit logs to capture all actions taken by any individual with administrative access. Administrative actions are high impact and must be traceable.

Event Detail
A log event without enough context may be useless during an investigation. Required fields should allow investigators to correlate events across systems and distinguish one user, component, or event from another.

PCI DSS v4.0 Requirement 10.2.2 requires audit log entries to include user identification, type of event, date and time, success or failure indication, event origination, and the identity or name of affected data, system component, resource, or service.

PCI DSS v4.0 Requirement 10.2.1.5 requires audit logs to capture all changes to identification and authentication credentials, including creation of new accounts and elevation of privileges. These events help detect unauthorized account manipulation.

PCI DSS v4.0 Requirement 10.2.1.7 requires audit logs to capture creation and deletion of system-level objects. Such events can indicate unauthorized changes to operating systems, databases, or applications.

Log Protection
Logs are valuable only if they can be trusted. Attackers often try to erase or alter logs after gaining access. Access to logs should be restricted, and changes should be detected or prevented.

PCI DSS v4.0 Requirement 10.3.1 requires read access to audit logs to be limited to those with a job-related need. Log access should not be broadly available.

PCI DSS v4.0 Requirement 10.3.2 requires audit log files to be protected from unauthorized modifications. The entity must prevent or detect changes that could undermine log integrity.

PCI DSS v4.0 Requirement 10.3.3 requires audit log files, including those for external-facing technologies, to be promptly backed up to a secure, centralized, internal log server or other media that is difficult to modify.

Log Review and Alert Handling
Log review should focus on systems and events most likely to show compromise or control failure. Automated alerting can help, but the entity must still define what is reviewed, how often it is reviewed, and how anomalies are handled.

PCI DSS v4.0 Requirement 10.4.1 requires daily review of audit logs for security events, logs of system components that store, process, or transmit cardholder data or sensitive authentication data, logs of critical system components, and logs of servers and system components that perform security functions.

PCI DSS v4.0 Requirement 10.4.2 requires logs of all other system components to be reviewed periodically. The review frequency must be appropriate to the risk of the systems and events.

PCI DSS v4.0 Requirement 10.4.3 requires exceptions and anomalies identified during log review to be addressed. Review without follow-up does not satisfy the purpose of the control.

Retention and Time Synchronization
Investigations often require historical logs. If logs are missing, overwritten, or stored with inconsistent timestamps, incident response becomes slower and less reliable. Retention and time synchronization therefore support both monitoring and forensic reconstruction.

PCI DSS v4.0 Requirement 10.5.1 requires audit log history to be retained for at least 12 months, with at least the most recent three months immediately available for analysis. Older logs may be archived if they can be retrieved when needed.

PCI DSS v4.0 Requirement 10.6.1 requires system clocks and time to be synchronized using time-synchronization technology. Consistent time settings allow events from different systems to be correlated.

PCI DSS v4.0 Requirement 10.7.1 requires failures of critical security control systems to be detected, alerted, and promptly responded to. Examples include failures of logging, intrusion detection, change detection, anti-malware, access controls, and segmentation controls.

Operational Evidence
Evidence for Requirement 10 includes logging configurations, SIEM rules, samples of events with required fields, log access permissions, file integrity controls, centralized log forwarding settings, daily review records, anomaly tickets, retention settings, and time synchronization configurations.
