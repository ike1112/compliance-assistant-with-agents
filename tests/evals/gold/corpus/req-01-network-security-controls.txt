PCI DSS Requirement 1: Network Security Controls

Scope and Intent
Requirement 1 addresses how an entity designs, implements, and maintains network security controls that protect the cardholder data environment from untrusted networks and unnecessary traffic. These controls include firewalls, router rules, cloud security groups, host-based filtering, and other technologies that enforce network traffic restrictions. The purpose is not merely to install a firewall; the entity must know what traffic is allowed, why it is allowed, where it is documented, and how changes are controlled.

Roles, Documentation, and Rule Management
Network security control standards must define how controls are configured, maintained, and reviewed for all connections into and out of the cardholder data environment. The standards should identify responsible personnel, permitted services, secure management methods, and the evidence needed to show that rule sets are intentional rather than accidental.

PCI DSS v4.0 Requirement 1.2.1 expects configuration standards for network security controls to be defined, implemented, and maintained. The standards must support consistent rule management across firewalls, routers, virtual network controls, and other technologies used to restrict traffic.

PCI DSS v4.0 Requirement 1.2.5 requires all allowed services, protocols, and ports to be identified, approved, and tied to a defined business need. If an insecure service, protocol, or port is used, the entity must implement security features that reduce the risk of that use.

PCI DSS v4.0 Requirement 1.2.7 requires reviews of network security control configurations at least once every six months. The review should confirm that rules remain justified, documented, and aligned with current business and security needs.

Network Diagrams and Data Flow Diagrams
An assessor or engineer should be able to understand how cardholder data moves without reverse-engineering the environment from device configurations. Diagrams help show boundaries, segmentation points, connections to third parties, wireless networks, and data flows that can change the scope of the assessment.

PCI DSS v4.0 Requirement 1.2.3 requires an accurate network diagram showing connections between the cardholder data environment and other networks, including wireless networks. The diagram must be kept current so it remains useful for scope and control validation.

PCI DSS v4.0 Requirement 1.2.4 requires a current data-flow diagram showing movement of account data across systems and networks. The diagram should identify where account data enters, is processed, is stored, and leaves the environment.

Inbound and Outbound Traffic
Traffic restrictions should be explicit. Only traffic needed for business or technical operations should be allowed, and all other traffic should be denied. This applies to inbound traffic from untrusted networks and outbound traffic from sensitive systems.

PCI DSS v4.0 Requirement 1.3.1 requires inbound traffic to the cardholder data environment to be restricted to traffic that is necessary and specifically authorized. The requirement supports a default-deny model for inbound connectivity.

PCI DSS v4.0 Requirement 1.3.2 requires outbound traffic from the cardholder data environment to be restricted to traffic that is necessary and specifically authorized. This prevents compromised systems from freely communicating with unauthorized destinations.

PCI DSS v4.0 Requirement 1.3.3 requires network security controls between all wireless networks and the cardholder data environment, regardless of whether the wireless network is considered in scope. The controls must deny or explicitly permit traffic based on documented business need.

Public Access and the DMZ
Systems that are reachable from the Internet create a higher-risk path toward account data. Public-facing components should be placed in a controlled network segment that limits direct connectivity to the cardholder data environment. The design should prevent a public host from becoming a simple bridge into systems that store or process cardholder data.

PCI DSS v4.0 Requirement 1.4.1 requires controls between untrusted networks and system components in the cardholder data environment. The controls must limit connectivity so only authorized traffic can reach in-scope systems.

PCI DSS v4.0 Requirement 1.4.2 requires inbound traffic from untrusted networks to be restricted to system components that are authorized to provide publicly accessible services, protocols, and ports. Public systems should not create broad paths to internal cardholder data systems.

PCI DSS v4.0 Requirement 1.4.4 requires system components that store cardholder data not to be directly accessible from untrusted networks. The intent is to prevent Internet-originated connections from reaching stored account data repositories.

Administration and Anti-Spoofing
Network security controls are themselves sensitive systems. Administrative access to them should be secured, and traffic that appears to use forged or private source addresses should be blocked when it enters from an untrusted side.

PCI DSS v4.0 Requirement 1.2.6 requires security features to be defined and implemented for insecure services, protocols, or ports that are allowed. The compensating security features must be appropriate to the risk of the allowed traffic.

PCI DSS v4.0 Requirement 1.4.3 requires anti-spoofing measures to detect and block forged source IP addresses from entering the network. This includes blocking traffic from untrusted networks that claims to originate from internal or otherwise unauthorized addresses.

PCI DSS v4.0 Requirement 1.5.1 requires security policies and operational procedures for Requirement 1 to be documented, kept current, used, and known to affected personnel. Operational knowledge matters because network controls must be maintained over time.

Operational Evidence
Good evidence for Requirement 1 includes approved rule requests, current diagrams, configuration exports, change tickets, periodic review records, and examples showing that denied traffic is actually blocked. The strongest evidence ties each permitted path to a business owner and a current technical need.
